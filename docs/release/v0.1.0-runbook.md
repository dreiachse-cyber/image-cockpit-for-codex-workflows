# v0.1.0 Release Runbook

更新日: 2026-06-23

This runbook is the confirmation-gated path from the private release candidate to the first public release. Do not merge `main`, change repository visibility, or create a public release until the owner explicitly approves those actions.

## Current Candidate

- Candidate: `0.1.0-rc.1`
- Branch: `codex/mvp-local-cockpit`
- Draft PR: https://github.com/dreiachse-cyber/image-cockpit-for-codex-workflows/pull/1
- Repository visibility: private until owner approval
- Review report: `docs/review/mvp-review-report.md`
- Checklist: `docs/release/v0.1.0-checklist.md`

## Local Review

1. Start the app:

   ```powershell
   npm run dev:all
   ```

2. Open the printed local URL.
3. Confirm Guided Start shows the four primary workflows.
4. Create an image generation Codex handoff job and confirm it is prompt-focused.
5. Create an image editing Codex handoff job and confirm it includes selected source asset, edit notes, and annotations.
6. If `codex exec` is unavailable, confirm the runner badge and job status show `unavailable` instead of waiting forever.
7. Place a PNG/WebP/GIF in `codex-handoff/outbox/`, then import it through Local Inbox.
8. Split the image into sprite frames and confirm the timeline is populated.
9. In Sprite Sheet Editing, check frame size, anchor, chroma key cleanup, and exports.

## Required Commands

Run these before approving a release candidate:

```powershell
npm run typecheck
npm test
npm run build
npm run smoke
```

Expected result: all commands pass.

## Codex Runner Verification

Run the local API and check runner preflight:

```powershell
Invoke-RestMethod http://127.0.0.1:8787/api/codex/runner
```

Acceptable private MVP outcomes:

- `ready`: the local Codex executable can be launched from the handoff server.
- `disabled`: manual handoff mode is active.
- `unavailable`: the job should still be written locally and the UI must not wait forever.

Before public release, verify `codex exec` on at least one environment where the configured Codex command can run without `spawn EPERM`, or explicitly release with manual handoff documented as the supported first-run path.

## Approval Gates

Do not proceed past each gate until it is explicitly approved.

1. Owner reviews `docs/review/mvp-review-report.md`.
2. Release notes in `CHANGELOG.md` are approved.
3. The final private PR diff is approved.
4. Main merge is approved.
5. Repository visibility change to public is approved.
6. Release tag and GitHub release creation are approved.

## Public Release Steps

Only after the approval gates above:

1. Merge the approved PR into `main`.
2. Confirm `main` is green with the required commands.
3. Create an annotated tag:

   ```powershell
   git tag -a v0.1.0 -m "v0.1.0"
   git push origin v0.1.0
   ```

4. Change repository visibility from private to public.
5. Create the GitHub release from `CHANGELOG.md`.
6. Confirm README links render correctly on the public repository.

## Do Not Ship If

- A direct OpenAI API call or API key requirement was added.
- `codex-handoff/`, `.env`, generated outputs, model weights, or license-unclear assets are staged.
- Guided Start no longer exposes all four primary workflows.
- Codex job creation can be clicked repeatedly while a pending job is waiting.
- Local Inbox cannot import an outbox image.
- The owner has not approved public visibility or main merge.
